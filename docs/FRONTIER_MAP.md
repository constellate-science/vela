# Frontier maps and knowledge dark matter

## 1. Dark matter is typed missingness

The phrase `knowledge dark matter` is useful only when it is operational.

A dark-matter object is an open `Obligation` generated by a named adapter version. It contains:

```text
obligation id
adapter and generator ids
target cell and context
gap kind
discharge evaluator
required verifier profile
dependencies and rationale
base presentation and view roots
```

The system can therefore answer:

```text
what is missing
why the system believes it is missing
what would discharge it
which accepted state and policy view produced the conclusion
```

## 2. Gap classes

The common gap vocabulary is deliberately small.

| Class | Meaning |
|---|---|
| `coverage` | A declared parameter, context, case, or benchmark cell lacks support. |
| `dependency` | An accepted route is blocked by an unsupported premise. |
| `discord` | Support and refutation or incompatible confidence coexist. |
| `translation` | A bridge candidate lacks its target-domain receipt. |
| `replication` | A policy requires an additional independent route. |
| `robustness` | Active support depends on too few or too fragile environments. |
| `model_residual` | A registered model leaves a declared residual or prediction obligation. |
| `failure_memory` | A search region has tried routes but no accepted discharge. |

Domain adapters may add subtypes. They may not redefine the core state semantics.

> Note (v0.700): `translation` and `model_residual` describe gaps against the cross-domain bridge and
> the model/prediction object families, which were retired. They remain in the vocabulary as a
> general taxonomy but are not minted by the current math-wedge frontiers.

## 3. Coverage models

A useful frontier map requires a declared coverage model. Examples include:

```text
all n in a registered Sidon interval
all proof obligations emitted by a Lean declaration
all parameter cells in a PDE design grid
all tests in a frozen software suite
all registered estimands in an analysis plan
all protocol arms and endpoints in an experiment plan
all target evaluations required by a model card
```

Coverage models are scientific commitments. They are versioned and reviewable. A broad model exposes more gaps but may create low-value work. A narrow model may hide important missingness. The map must show the chosen boundary.

## 4. Unknown unknowns

No frontier system can infer a complete set of unknown unknowns from accepted state alone. The same visible state is compatible with many possible obligation universes.

Constellate can expand its visible frontier by:

- admitting new adapters and coverage schemas;
- mining contradictions and unresolved dependencies;
- importing registered research programs;
- proposing relation templates from graph or language models;
- learning residual structures from data or simulations;
- accepting expert-declared missing dimensions.

These operations generate candidate obligations. They do not prove that the map is complete.

## 5. Frontier migration

An obligation has three states:

```text
latent       its prerequisite cells are not yet active
open         its prerequisites are active and its target is unsupported
discharged   its target is actively supported
```

This gives Constellate a moving frontier rather than a static checklist. Under a
fixed view and positive append, a discharged monotone-support obligation cannot
reopen. An append may instead expose a successor by moving it from `latent` to
`open`. A restriction may reverse that movement because active support has
changed without erasing historical lineage.

For a parameter sequence this can read:

```text
before:  alpha=.2 open,        alpha=.3 latent
after:   alpha=.2 discharged,  alpha=.3 open
restrict alpha=.2 receipt:
         alpha=.2 open,        alpha=.3 latent
repair:  alpha=.2 discharged,  alpha=.3 open
```

This is the operational meaning of extending a frontier. The boundary of
addressable work moves outward as accepted state grows.

## 6. Frontier quality

A frontier map should be evaluated on:

```text
precision     experts judge surfaced obligations to be real
recall        known obligations are represented
freshness     accepted updates change the map promptly
utility       producers choose work differently because of the map
closure       discharged obligations disappear by replay, not manual editing
inheritance   the next producer begins from the current map
```

Schema count is not a quality metric.

## 7. Ranking is not trust

Constellate may rank obligations using:

```text
structural dependents unlocked
expected information gain
verification or experiment cost
probability of success
transfer reuse
failure cost avoided
safety and access constraints
```

The ranking packet must name its model, priors, utility, costs, and base roots. A rank may schedule work. It cannot change support, evidence class, confidence, or trust coordinates.

## 8. Example

The numerical-simulation adapter declares a parameter frontier. The accepted
presentation initially supports `alpha = 0.1`. The `0.2` obligation is actionable;
`0.3` is declared but depends on `0.2` and is therefore latent.

```text
open(P)   = {solution(alpha=.2)}
latent(P) = {solution(alpha=.3)}
```

A neural operator proposes samples for `0.2`. The map does not change. A target
semantic-replay receipt passes and a human accepts the route. The `0.2`
obligation becomes discharged and `0.3` becomes the new open frontier. A
challenge to the target receipt moves the map back. An independently replayed
repair moves it forward again.
